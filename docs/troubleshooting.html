Common problems & fixes

Kiosk service runs but nothing appears

Symptom. systemctl status speculum.service shows active (or it starts and immediately "Deactivated successfully" after ~1 second), but the mirror never draws on the display.

Cause. cage needs to own the seat (the GPU / tty1). If the Pi boots into the desktop, the desktop session already holds seat0, so cage gets no surface and the app exits cleanly with no output.

Fix — boot to console so the kiosk owns the screen:

systemctl get-default                       # graphical.target = boots to desktop (the conflict)

sudo systemctl set-default multi-user.target   # boot to console, no desktop
sudo loginctl enable-linger "$USER"            # create /run/user/<uid> at boot (see below)
sudo systemctl enable speculum.service
sudo reboot

After reboot, with no desktop competing, cage takes tty1 and the mirror renders fullscreen. Check:

systemctl status speculum.service --no-pager
journalctl -u speculum.service --since today -n 30 --no-pager
command -v cage      # must be installed:  sudo apt install cage
To watch the real failure live, switch to a bare console (Ctrl+Alt+F1, log in), sudo systemctl stop speculum.service, then run cage -s -- /opt/speculum/bin/speculum and read what it prints.

"XDG_RUNTIME_DIR is not set in the environment"

Symptom. The service crash-loops with cage[…] [ERROR] XDG_RUNTIME_DIR is not set in the environment.

Cause. cage needs a user runtime dir (/run/user/<uid>). The unit asks pam_systemd (PAMName=speculum) to create it, but with no active login session there's nothing to attach to.

Fix — enable lingering so systemd creates the runtime dir at boot:

sudo loginctl enable-linger "$USER"
sudo systemctl restart speculum.service

If it still complains, pin it explicitly:

sudo systemctl edit speculum.service
# add:
#   [Service]
#   Environment=XDG_RUNTIME_DIR=/run/user/1000      # 1000 = id -u <your-user>
sudo systemctl daemon-reload && sudo systemctl restart speculum.service

Autologin isn't needed for the service

For the packaged speculum.service kiosk you do not need console autologin. The unit opens its own PAM session bound to tty1/seat0 and cage takes the screen directly. Trying to also set up autologin is a different, competing launch model — don't mix them. If you'd rather launch the mirror from an autologin shell instead of the service, run it bare in your session and disable speculum.service (then updates use the session self-relaunch path).

Two mirrors / port 8080 in use

Symptom. The admin UI / curl localhost:8080/api/version reports an old version even after a successful update; pgrep -af speculum shows a process that predates the update.

Cause. Two independent mirrors are running — e.g. a desktop-session launch and an enabled speculum.service. They both try to bind port 8080; whichever started first holds it, and the updater may restart the other one (the one not on screen).

Fix — run exactly one launcher. For a kiosk, boot to console and use only the service (above); kill any stray session mirror:

pgrep -af speculum                       # see every instance
sudo systemctl disable --now speculum.service   # …if you instead want the desktop launch
pkill -f /opt/speculum/bin/speculum             # stop a stale session mirror
curl -s localhost:8080/api/version              # confirm one instance, right version

Updater says "install manually" / "release is unsigned"

Cause. The in-app updater is fail-closed: it only installs a release whose SHA256SUMS carries a valid GPG signature from the key bundled in the app. If the release has no SHA256SUMS.asc (signing not configured in CI), it refuses and points you at the releases page. Development builds also show this.

Fix. Update manually for now. For maintainers: configure the GPG_PRIVATE_KEY / GPG_PASSPHRASE repo secrets (matching KEYS) and cut a release — the release workflow then signs every asset and the updater goes live automatically.

"Couldn't check GitHub for updates"

  • No network / rate-limited. The check uses the unauthenticated GitHub API. Verify outbound HTTPS to api.github.com.
  • Wrong repo. The default repo is the canonical pierrejochem/SpeculumSmartMirror. If you set a custom repo in the updatenotifier module config, make sure it points at a repo that publishes signed releases.

Update fails: permission denied / read-only / timeout

These were bugs fixed in v0.5.6; if you hit them, update to v0.5.6+ (manually once) and they go away:

Journal lineCause & fix
staged.pkg: Permission deniedOld builds staged into a root-owned dir tied to user pi. Now staged under the mirror user's own ~/.speculum/update. Fixed in v0.5.6.
Read-only file system creating the packageThe updater unit set ProtectHome=read-only, blocking the home-staging write. Removed in v0.5.6.
Request timeout has expired mid-downloadThe HTTP client's default 15 s request cap aborted the ~100 MB download. The downloader now has no whole-request timeout (only a stall timeout). Fixed in v0.5.6.
"could not authorize (polkit)"polkit isn't installed or the rule didn't load. Confirm polkit is present, or use the sudoers fallback. The rule authorizes any active local session to start speculum-update.service.

Inspect the privileged side any time with:

journalctl -u speculum-update.service -n 40 --no-pager

Updated, but the screen still shows the old version

Cause. The package installed, but the running mirror process wasn't the one that got restarted — usually the two-mirrors situation, or the on-screen mirror runs a different launcher than the one the helper restarted.

Fix. Settle on one launch method (above). The helper restarts speculum.service only when it's active; a --user service or a bare session launch is restarted by the app itself. The installed files are already the new version, so simply relaunching the on-screen mirror once brings it to the new release.
